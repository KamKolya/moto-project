# Система обліку та продажу мотоциклів

Повна технічна та бізнес-документація до навчального full-stack застосунку для обліку мотоциклів, оформлення замовлень, ведення клієнтської бази, перегляду звітів та адміністрування профілю.

Проєкт реалізований як моноліт із фронтендом на `React + Vite`, API на `Express` і локальною базою даних `SQLite` через `better-sqlite3`.

## 1. Призначення системи

Система вирішує п'ять основних задач:

1. Вести каталог мотоциклів і контролювати фактичні залишки на складі.
2. Оформлювати замовлення як для існуючих, так і для нових клієнтів.
3. Автоматично резервувати та повертати складські залишки залежно від статусу замовлення.
4. Агрегувати історію клієнта: кількість замовлень, суму покупок, статус активності.
5. Формувати аналітику продажів і ризикових залишків.

Система орієнтована на один адміністративний інтерфейс, без ролей доступу на рівні окремих екранів. Є лише один активний профіль адміністратора, який одночасно є джерелом авторизаційної ідентичності.

## 2. Технологічний стек

| Шар | Технології | Призначення |
| --- | --- | --- |
| UI | `React 19`, `Vite`, `lucide-react` | Адмін-панель, форми, таблиці, локальні модальні стани |
| Стилі | `Tailwind CSS 4` | Візуальний інтерфейс |
| API | `Express 4`, `zod` | REST API, валідація запитів, auth middleware |
| База даних | `SQLite`, `better-sqlite3` | Зберігання каталогу, замовлень, клієнтів, профілю |
| Спільний домен | `shared/domain.ts` | Єдині enum-значення, переходи статусів, нормалізація довідників |
| Тести | `Vitest`, `supertest` | Unit, integration, smoke E2E |

## 3. Архітектура

### 3.1. Загальна схема

```text
React UI (src/)
   -> src/api.ts
   -> /api/*
Express API (server/index.ts)
   -> валідація Zod
   -> бізнес-логіка server/db.ts
   -> SQLite (data/app.db)
```

### 3.2. Шари відповідальності

- `src/` відповідає за інтерфейс, локальні стани форм, показ toast-повідомлень і повторне завантаження даних після кожної мутації.
- `server/index.ts` відповідає за HTTP-маршрути, авторизацію, валідацію `zod` і перетворення помилок у дружні відповіді API.
- `server/db.ts` містить головну бізнес-логіку: схему БД, сидування, міграції, нормалізацію даних, CRUD-операції, транзакції, звіти та обмеження цілісності.
- `shared/domain.ts` задає спільні правила між клієнтом і сервером: статуси замовлень, дозволені переходи, категорії мотоциклів, сегменти клієнтів та їх нормалізацію.

### 3.3. Ключовий принцип синхронізації

Фронтенд не підтримує складну локальну синхронізацію після мутацій. Після створення, оновлення чи видалення будь-якої сутності він знову запитує:

- `/api/bootstrap`
- `/api/reports/overview`

Тобто джерелом істини завжди є сервер і БД, а не клієнтський state.

## 5. Доменна модель і зв'язки

### 5.1. Сутності

#### `inventory`

Каталог моделей мотоциклів і складський залишок.

| Поле | Тип | Значення |
| --- | --- | --- |
| `id` | `TEXT` | Бізнес-ID, формат на кшталт `M001`, `W001` |
| `name` | `TEXT` | Назва моделі |
| `category` | `TEXT` | Категорія з фіксованого довідника |
| `year` | `TEXT` | Рік моделі у форматі `YYYY` |
| `engine` | `TEXT` | Технічний опис двигуна |
| `price_cents` | `INTEGER` | Ціна в центах USD |
| `stock` | `INTEGER` | Доступний до продажу залишок, уже без активних резервів |
| `status` | `TEXT` | Людинозрозумілий складський статус |
| `image` | `TEXT` | URL або `data:` URL зображення |
| `created_at` | `TEXT` | Технічний ISO timestamp |

Важлива деталь: у відповіді API для складу повертаються ще два похідні поля:

- `reservedStock` = кількість одиниць, зарезервованих активними замовленнями
- `onHandStock` = `stock + reservedStock`

Тобто:

- `stock` = вільний залишок
- `onHandStock` = загальна фізична наявність

#### `orders`

Замовлення в цій системі однотоварні: одне замовлення містить один товарний запис `inventory_id` і одну кількість `quantity`.

| Поле | Тип | Значення |
| --- | --- | --- |
| `id` | `TEXT` | Бізнес-ID, формат `ORD-001` |
| `customer` | `TEXT` | Денормалізована назва клієнта |
| `phone` | `TEXT` | Денормалізований телефон клієнта |
| `date` | `TEXT` | Бізнес-дата у форматі `YYYY-MM-DD`, формується в часовому поясі `Europe/Kyiv` |
| `total_cents` | `INTEGER` | Сума замовлення в центах |
| `status` | `TEXT` | Поточний статус замовлення |
| `items` | `TEXT` | Текстовий опис на кшталт `2x Ducati Panigale V4` |
| `quantity` | `INTEGER` | Кількість одиниць |
| `inventory_id` | `TEXT` | FK на `inventory.id` |
| `customer_id` | `TEXT` | FK на `customers.id` |
| `created_at` | `TEXT` | Технічний ISO timestamp |

Замовлення зберігає і зовнішні ключі, і денормалізовані поля. Це потрібно для зручного відображення в UI та збереження текстового контексту.

#### `customers`

Клієнтська база.

| Поле | Тип | Значення |
| --- | --- | --- |
| `id` | `TEXT` | Бізнес-ID, формат `CUST-001` |
| `name` | `TEXT` | ПІБ або назва компанії |
| `phone` | `TEXT` | Канонічний телефон, унікальний |
| `email` | `TEXT` | Email, унікальний |
| `segment` | `TEXT` | Сегмент: `Стандарт`, `VIP`, `Оптовий` |
| `total_orders` | `INTEGER` | Кількість нескасованих замовлень |
| `total_spent_cents` | `INTEGER` | Сума лише виконаних замовлень |
| `status` | `TEXT` | Автоматично обчислюваний статус: `Новий` або `Активний` |
| `created_at` | `TEXT` | Технічний ISO timestamp |

#### `profile`

Єдиний профіль адміністратора.

| Поле | Тип | Значення |
| --- | --- | --- |
| `id` | `INTEGER` | Завжди `1` |
| `name` | `TEXT` | Ім'я адміністратора |
| `role` | `TEXT` | Підпис ролі в UI і токені |
| `email` | `TEXT` | Email для входу |
| `avatar` | `TEXT` | URL або `data:` URL аватара |
| `session_version` | `INTEGER` | Версія сесії для інвалідації токенів |
| `updated_at` | `TEXT` | Технічний ISO timestamp |

### 5.2. Зв'язки між сутностями

| Зв'язок | Тип | Правило |
| --- | --- | --- |
| `inventory -> orders` | `1:N` | Один товар може бути в багатьох замовленнях |
| `customers -> orders` | `1:N` | Один клієнт може мати багато замовлень |
| `profile` | `1:1` | Окрема singleton-сутність |

SQLite FK:

- `orders.inventory_id -> inventory.id`
- `orders.customer_id -> customers.id`

Політика видалення:

- товар не можна видалити, якщо з ним пов'язані замовлення
- клієнта не можна видалити, якщо з ним пов'язані замовлення
- замовлення не можна видалити, якщо воно вже виконане

### 5.3. Денормалізовані поля

У системі є навмисна денормалізація:

- `orders.customer` і `orders.phone` дублюють дані клієнта
- `orders.items` дублює текстову назву товару

При оновленні клієнта система синхронізує:

- `customers.name`
- `customers.phone`
- усі `orders.customer`
- усі `orders.phone`

При оновленні товару назва в старих `orders.items` не переписується. Отже:

- клієнтські контакти поводяться як синхронізований денормалізований шар
- товарна назва в `orders.items` поводиться як текстовий знімок на момент створення замовлення

## 6. Довідники та бізнес-правила

### 6.1. Категорії мотоциклів

Дозволені категорії:

`Адвенчер`, `Боббер`, `Електро`, `Ендуро`, `Інше`, `Кафе-рейсер`, `Класик`, `Крос`, `Круїзер`, `Міський`, `Мопед`, `Нейкед`, `Роадстер`, `Скутер`, `Скремблер`, `Спорт`, `Спорт-турер`, `Турер`, `Чопер`

Система вміє нормалізувати низку псевдонімів:

- `commuter`, `комютер`, `ком'ютер`, `комутер` -> `Міський`
- `дорожній` -> `Роадстер`
- `класичний` -> `Класик`
- `пригодницький` -> `Адвенчер`
- `спорт турист`, `спорт-турист` -> `Спорт-турер`

На API-рівні некоректні категорії відхиляються. У міграціях legacy-значення перетворюються, а невідомі категорії переводяться в `Інше`.

### 6.2. Сегменти клієнтів

Дозволені сегменти:

- `Стандарт`
- `VIP`
- `Оптовий`

Підтримуються псевдоніми:

- `vip` -> `VIP`
- `опт`, `wholesale` -> `Оптовий`

### 6.3. Статуси клієнтів

Статуси клієнтів не вводяться вручну користувачем. Вони завжди обчислюються із замовлень:

- `Новий` -> якщо `total_orders = 0`
- `Активний` -> якщо `total_orders > 0`

`total_orders` враховує всі нескасовані замовлення.

`total_spent_cents` враховує лише виконані замовлення.

Отже клієнт може стати `Активний`, навіть якщо ще не має жодного виконаного замовлення, але вже має хоча б одне замовлення в роботі або в очікуванні оплати.

### 6.4. Статуси товарного залишку

Статус товару формується автоматично з поточного доступного залишку `stock`:

- `Немає` -> `stock <= 0`
- `Закінчується` -> `stock <= 3`
- `В наявності` -> `stock > 3`

### 6.5. Статуси замовлень

Дозволені значення:

- `В обробці`
- `Очікує оплати`
- `Виконано`
- `Скасовано`

Дозволені переходи:

| Поточний статус | Доступні наступні статуси |
| --- | --- |
| `В обробці` | `Очікує оплати`, `Виконано`, `Скасовано` |
| `Очікує оплати` | `В обробці`, `Виконано`, `Скасовано` |
| `Виконано` | переходи заборонені |
| `Скасовано` | `В обробці`, `Очікує оплати` |

### 6.6. Які статуси резервують склад

Замовлення резервує склад, якщо його статус:

- `В обробці`
- `Очікує оплати`

Замовлення не резервує склад, якщо його статус:

- `Виконано`
- `Скасовано`

## 7. Життєвий цикл даних

### 7.1. Створення товару

При створенні товару система:

1. Генерує новий ID `M###`.
2. Нормалізує і перевіряє категорію.
3. Перевіряє рік у діапазоні `1900..(поточний рік + 1)`.
4. Парсить ціну в центах.
5. Присвоює `stock` як доступний залишок.
6. Автоматично визначає складський статус.

### 7.2. Оновлення товару

Найважливіша особливість цієї системи: під час оновлення товару поле `stock` у payload інтерпретується як бажаний загальний фізичний залишок (`onHandStock`), а не як вільний залишок.

Алгоритм:

1. Сервер обчислює `reservedStock` для товару.
2. Перевіряє, що новий загальний залишок не менший за резерв.
3. Вираховує `stock = onHandStock - reservedStock`.
4. Оновлює рядок у БД.

Це означає:

- у фронтенді для редагування потрібно підставляти `onHandStock`
- якщо зовнішній клієнт API відправить `stock`, рівний лише вільному залишку, то загальний фізичний залишок фактично зменшиться

### 7.3. Створення замовлення

Алгоритм `createOrder`:

1. Перевіряється існування товару.
2. Перевіряється, що `inventory.stock >= quantity`.
3. Визначається клієнт:
   - або вибирається існуючий
   - або створюється новий клієнт із сегментом `Стандарт`
4. Формується новий ID `ORD-###`.
5. `total_cents = inventory.price_cents * quantity`.
6. `date` виставляється як поточна дата в часовому поясі `Europe/Kyiv`.
7. Нове замовлення завжди створюється зі статусом `В обробці`.
8. Залишок товару зменшується на `quantity`.
9. Для клієнта перераховуються `total_orders`, `total_spent_cents`, `status`.

### 7.4. Новий клієнт через замовлення

Якщо замовлення створюється в режимі `customerMode = new`, система:

- створює нового клієнта тільки якщо телефон і email унікальні
- не виконує "тихого" пошуку існуючого клієнта по телефону
- повертає `409 Conflict`, якщо телефон або email уже зайняті

### 7.5. Оновлення статусу замовлення

При зміні статусу система враховує вплив на склад:

| Перехід | Ефект на склад |
| --- | --- |
| `В обробці -> Очікує оплати` | без змін |
| `Очікує оплати -> В обробці` | без змін |
| `В обробці -> Виконано` | без змін, бо товар уже був зарезервований |
| `Очікує оплати -> Виконано` | без змін |
| `В обробці -> Скасовано` | резерв повертається на склад |
| `Очікує оплати -> Скасовано` | резерв повертається на склад |
| `Скасовано -> В обробці/Очікує оплати` | резерв накладається знову, але лише якщо вистачає залишку |

### 7.6. Видалення замовлення

Правила:

- `Виконано` видаляти не можна
- якщо замовлення було в статусі, що резервує склад, резерв повертається
- якщо замовлення було `Скасовано`, повторного повернення не відбувається
- після видалення замовлення клієнтські агрегати перераховуються

### 7.7. Оновлення клієнта

При оновленні клієнта:

1. Перевіряється унікальність телефону та email.
2. Телефон нормалізується до канонічної форми.
3. Оновлюється сам клієнт.
4. Синхронізуються `orders.customer` і `orders.phone`.
5. Статус клієнта знову обчислюється з історії замовлень.

Важливо: зміна сегмента не впливає на статус. Сегмент і статус живуть окремо.

### 7.8. Оновлення профілю

При оновленні профілю:

- `name` і `avatar` змінюються без інвалідації поточної сесії
- `email` або `role` збільшують `session_version`
- старі токени після цього стають невалідними
- новий вхід можливий лише з оновленим email

## 8. Правила валідації та нормалізації

### 8.1. Телефон

Телефон нормалізується так:

- `0XXXXXXXXX` -> `+38XXXXXXXXXX`
- `380XXXXXXXXX` -> `+380XXXXXXXXX`
- будь-який інший цифровий набір -> `+<digits>`

Отже `+380991112233` і `+38 (099) 111-22-33` вважаються одним і тим самим значенням.

### 8.2. Ціни

Сервер приймає ціну як `string | number`.

Підтримуються формати:

- `19800`
- `19800.50`
- `19 800`
- `$19,800.50`
- `19,800.50`

Негативні ціни заборонені.

У БД гроші зберігаються в `*_cents`.

API повертає гроші у двох різних формах:

- для складу, замовлень і клієнтів додатково повертаються готові рядки, сформовані сервером (`$19,800.50`)
- для звітів сервер повертає числові значення в major units, а UI форматує їх уже на клієнті через `Intl.NumberFormat('uk-UA', { currency: 'USD' })`

### 8.3. Рік моделі

На API рівні рік має відповідати regex `^\d{4}$`.

На рівні БД і бізнес-логіки:

- рік має бути від `1900`
- максимум `поточний рік + 1`

У legacy-міграціях система також уміє витягувати рік із дат на кшталт `DD/MM/YYYY` чи `YYYY-MM-DD`.

## 9. Ініціалізація БД, схема, індекси, міграції

### 9.1. Технічні параметри SQLite

Під час старту сервера:

- шлях до БД обчислюється з `DB_PATH`
- якщо шлях відносний, він резолвиться від `process.cwd()`
- каталог БД створюється автоматично
- встановлюється `journal_mode = WAL`
- вмикаються `foreign_keys = ON`

### 9.2. Схема та версії

У коді є:

- `STRUCTURE_VERSION = 6`
- `SCHEMA_VERSION = 9`

Використовується `PRAGMA user_version`.

Повноцінної зовнішньої системи міграцій немає. Всі міграції живуть у `server/db.ts` і виконуються імперативно при старті.

### 9.3. Strict schema

Таблиці створюються з жорсткими `CHECK`-обмеженнями:

- непорожні `id`, `name`, `engine`, `image`
- `category` лише зі списку дозволених категорій
- `year` лише чотири цифри
- `price_cents >= 0`
- `stock >= 0`
- `quantity > 0`
- унікальні `customers.phone` і `customers.email`
- `profile.id = 1`
- `session_version >= 1`

### 9.4. Індекси

Створюються:

- `idx_orders_inventory_id`
- `idx_orders_customer_id`
- `idx_customers_email_unique`

### 9.5. Автоматичні repair-процедури

На старті система не лише створює схему, а й ремонтує дані:

- додає відсутні колонки старим таблицям
- нормалізує сегменти клієнтів
- переводить email в lower-case
- канонізує телефони
- синхронізує телефони клієнтів у замовленнях
- додає/ремонтує `session_version` профілю
- виправляє аномальні масштаби грошей, якщо старі дані були помилково збережені не в тих одиницях
- перераховує клієнтські агрегати
- перераховує складські статуси
- створює індекси
- окремо виправляє відому стару аномалію з `ORD-003`

### 9.6. Legacy migration

Якщо знайдена застаріла структура, виконується повна міграція в нові strict-таблиці:

- створюються тимчасові `_new` таблиці
- старі дані вичитуються, очищуються, нормалізуються і вставляються заново
- при потребі створюються placeholder-товари
- клієнти відновлюються або зшиваються за phone/name
- після успішного перенесення старі таблиці замінюються новими

## 10. Початкове сидування

Якщо БД порожня, система створює:

- 25 товарів
  - 5 базових моделей `M001..M005`
  - 20 додаткових позицій `W001..W020`
- 3 клієнтів
- 3 замовлення
- 1 профіль адміністратора

Початкові замовлення потрібні не лише як демо-дані, а й як приклади різних станів:

- є виконані продажі
- є активне замовлення з резервом
- є приклад позиції з нульовим доступним залишком

Після сидування система одразу:

- застосовує резерви по активних замовленнях
- перераховує клієнтську статистику
- нормалізує складські статуси

## 11. Авторизація

### 11.1. Загальна логіка

Коли `AUTH_ENABLED=true`:

- публічні лише `/api/health` і `/api/auth/login`
- усі інші `/api/*` вимагають `Authorization: Bearer <token>`

Коли `AUTH_ENABLED=false`:

- клієнт не бачить екран входу
- API автоматично пропускає запити як поточний профіль

### 11.2. Формат токена

Токен не використовує сторонню JWT-бібліотеку. Це власний формат:

1. JSON payload із полями `email`, `role`, `sessionVersion`, `exp`
2. base64url-кодування payload
3. HMAC SHA-256 підпис через `AUTH_SECRET`

TTL токена: `12 годин`.

### 11.3. Захист від старих токенів

Під час валідації токена сервер перевіряє:

- термін дії
- коректність підпису
- актуальний `email`
- актуальну `role`
- актуальний `sessionVersion`

Тому зміна `email` або `role` у профілі автоматично "викидає" всі старі сесії.

### 11.4. Безпека дефолтних значень

Поза локальною розробкою сервер відмовиться запускатися з дефолтними:

- `AUTH_PASSWORD`
- `AUTH_SECRET`

Це захищає від випадкового деплою з тестовими обліковими даними.

## 12. API

### 12.1. Загальні правила

- Формат: JSON
- Валідація: `zod`
- Успішне видалення: `204 No Content`
- Типові помилки:
  - `400` -> некоректний payload або порушення бізнес-правил
  - `401` -> неавторизовано
  - `404` -> сутність не знайдена
  - `409` -> конфлікт унікальності або інша колізія

### 12.2. Ендпоїнти

#### `GET /api/health`

Публічний healthcheck.

Відповідь:

```json
{
  "ok": true,
  "timestamp": "2026-04-21T12:00:00.000Z"
}
```

#### `POST /api/auth/login`

Вхід за email і паролем.

Body:

```json
{
  "email": "admin@motosys.ua",
  "password": "your-password"
}
```

Response:

```json
{
  "token": "<bearer-token>",
  "user": {
    "email": "admin@motosys.ua",
    "role": "Адміністратор"
  }
}
```

#### `GET /api/auth/me`

Повертає стан авторизації і поточну identity.

```json
{
  "authEnabled": true,
  "user": {
    "email": "admin@motosys.ua",
    "role": "Адміністратор"
  }
}
```

#### `GET /api/bootstrap`

Головний endpoint для початкового завантаження UI.

Query params:

- `limit?: number` від `1` до `500`
- `offset?: number` від `0`

Важлива деталь: `limit/offset` застосовуються окремо до кожної таблиці `inventory`, `orders`, `customers`, а не до спільного потоку даних.

Response:

- `inventory[]`
- `orders[]`
- `customers[]`
- `profile`

Порядок:

- `ORDER BY created_at DESC, id DESC`

#### `GET /api/reports/overview`

Повертає агрегати для дашборда:

- summary
- statusStats
- topModels
- topCustomers
- lowStockItems

#### `POST /api/inventory`

Створення товару.

Body:

```json
{
  "name": "Suzuki GSX-8R",
  "category": "Спорт",
  "year": "2024",
  "engine": "776cc",
  "price": "9600",
  "stock": 3,
  "image": "https://..."
}
```

#### `PUT /api/inventory/:id`

Оновлення товару.

Body має ту саму структуру, що і створення, але `stock` трактується як бажаний загальний залишок `onHandStock`.

#### `DELETE /api/inventory/:id`

Видалення товару. Заборонено, якщо існують пов'язані замовлення.

#### `POST /api/orders`

Підтримує два режими.

Варіант 1: існуючий клієнт

```json
{
  "customerMode": "existing",
  "inventoryId": "M002",
  "customerId": "CUST-002",
  "quantity": 1
}
```

Варіант 2: новий клієнт

```json
{
  "customerMode": "new",
  "inventoryId": "M002",
  "customerName": "Новий клієнт",
  "phone": "+380991112233",
  "email": "new@example.com",
  "quantity": 1
}
```

#### `PUT /api/orders/:id/status`

Body:

```json
{
  "status": "Очікує оплати"
}
```

Дозволені лише статуси з доменного довідника і лише дозволені переходи.

#### `DELETE /api/orders/:id`

Видалення замовлення. Заборонено для `Виконано`.

#### `POST /api/customers`

Створення клієнта.

```json
{
  "name": "Moto Trade LLC",
  "phone": "+380501112244",
  "email": "trade@example.com",
  "segment": "VIP"
}
```

#### `PUT /api/customers/:id`

Оновлення клієнта. Після зміни імені або телефону дані оновляться і в таблиці замовлень.

#### `DELETE /api/customers/:id`

Видалення клієнта. Заборонено, якщо є пов'язані замовлення.

#### `PUT /api/profile`

Partial update профілю:

```json
{
  "name": "Менеджер",
  "role": "Менеджер з продажу",
  "email": "manager@motosys.ua",
  "avatar": "data:image/png;base64,..."
}
```

## 13. Звіти та їх бізнес-сенс

### 13.1. Summary

| Поле | Як рахується |
| --- | --- |
| `totalOrders` | усі замовлення |
| `completedOrders` | лише `status = Виконано` |
| `totalRevenue` | сума `total_cents` лише для виконаних замовлень |

| `totalStockUnits` | сума `inventory.stock`, тобто лише доступний залишок |
| `totalCustomers` | кількість рядків у `customers` |

### 13.2. `topModels`

Бере лише виконані замовлення, групує по `inventory_id`, рахує:

- `soldUnits`
- `revenue`

Ліміт: `7`.

### 13.3. `topCustomers`

Бере лише виконані замовлення, з'єднує з `customers`, рахує:

- кількість завершених замовлень
- суму завершених покупок

Ліміт: `7`.

### 13.4. `statusStats`

Розподіл усіх замовлень по статусах.

### 13.5. `lowStockItems`

Товари з доступним залишком `<= 3`, відсортовані за зростанням залишку.

## 14. Поведінка фронтенду

### 14.1. Загальний UI

- Односторінковий застосунок без роутера.
- Навігація через вкладки: `Склад`, `Замовлення`, `Клієнти`, `Звіти`, `Профіль`.
- Верхня панель показує узагальнені метрики.
- Toast-повідомлення використовуються для всіх успішних дій і помилок.

### 14.2. Авторизація

- При старті UI викликає `/api/auth/me`.
- Якщо токена немає, але `AUTH_ENABLED=false`, екран логіна не показується.
- Якщо токен прострочений або втратив актуальність, користувач логаутиться.

### 14.3. Модуль складу

Підтримує:

- пошук по `id`, назві, категорії, двигуну
- фільтр по категорії
- фільтр по статусу
- створення і редагування товару
- валідацію року на клієнті
- модальне завантаження фото
- перегляд детальної картки товару
- видалення через confirm modal

Особливості:

- у формі редагування підставляється `onHandStock`
- локальний upload фото зберігається як `data:` URL
- розмір фото обмежений `3 MB`

### 14.4. Модуль замовлень

Підтримує:

- вибір товару лише з позицій, де `stock > 0`
- два режими клієнта: `existing` і `new`
- перегляд деталей замовлення і клієнта
- зміну статусу лише на дозволені стани
- видалення з блокуванням для завершених замовлень

### 14.5. Модуль клієнтів

Підтримує:

- пошук по `id`, імені, телефону, email, сегменту
- створення
- редагування
- видалення

У UI явно зазначено, що статус клієнта формується автоматично з історії замовлень.

### 14.6. Модуль профілю

Підтримує:

- зміну імені
- ролі
- email для входу
- аватара

Аватар також може зберігатися як `data:` URL, обмеження файлу: `3 MB`.

### 14.7. Модуль звітів

Показує:

- картки summary
- таблицю топ-моделей
- графічний розподіл статусів
- таблицю топ-клієнтів
- список ризикових залишків

## 18. Обмеження та важливі нюанси

1. Одне замовлення містить лише один товарний рядок і кількість, а не кошик із багатьма позиціями.
2. Складський агрегат `totalStockUnits` у звітах і верхній картці дашборда показує лише доступний, а не повний фізичний залишок.
3. Частина грошових полів приходить як форматовані рядки, частина як числа в major units.
4. Завантажені зображення зберігаються як `data:` URL у БД, що може збільшувати розмір SQLite-файлу.
5. Видалення товарів і клієнтів жорстко блокується за наявності будь-яких пов'язаних замовлень, навіть старих.
6. Профіль у системі один, окремого довідника користувачів або RBAC немає.
7. Міграції та data-repair живуть у коді, а не в окремому інструменті міграцій.

## 19. Короткий висновок по бізнес-логіці

Проєкт реалізує не просто CRUD, а цілісний операційний контур:

- склад з доступним і зарезервованим залишком
- замовлення як рушій змін у складі
- клієнт як агрегат історії продажів
- звіти, що рахуються безпосередньо з транзакційних даних
- профіль, пов'язаний з авторизацією

Головна бізнес-ідея системи: замовлення є центром доменної логіки. Саме воно одночасно впливає на:

- доступність товару
- статус і статистику клієнта
- фінансові звіти
- аналітику по моделях

Тому при будь-якому розширенні проєкту в першу чергу потрібно враховувати наслідки змін саме для життєвого циклу замовлення і правил резервування складу.
